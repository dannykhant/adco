# Queries: Baseline vs v2 Optimized (deepseekv4flashmysqlv2)

## How to read this

Each transaction section shows:
- **Baseline**: per-TPC-C-spec, one query at a time
- **v1**: the original `deepseekv4flashmysql` optimizations
- **v2**: further merges and batch rewrites in `deepseekv4flashmysqlv2`
- **Rationale**: which KB method was applied and why

The metric: **number of database round-trips per transaction call**.

---

## DELIVERY

### Application flow

Loop over 10 districts. Per district with a pending order:

1. `SELECT NO_O_ID FROM NEW_ORDER` → oldest pending order
2. `SELECT O_C_ID FROM ORDERS` → customer who placed it
3. `SELECT SUM(OL_AMOUNT) FROM ORDER_LINE` → line-item total
4. `DELETE FROM NEW_ORDER` → remove from queue
5. `UPDATE ORDERS SET O_CARRIER_ID` → mark delivered
6. `UPDATE ORDER_LINE SET OL_DELIVERY_D` → delivery timestamp
7. `UPDATE CUSTOMER SET C_BALANCE` → add amount to balance

### Baseline (10 × 7 = 70 round-trips)

```sql
SELECT NO_O_ID FROM NEW_ORDER
WHERE NO_D_ID = %s AND NO_W_ID = %s AND NO_O_ID > -1 LIMIT 1;

SELECT O_C_ID FROM ORDERS
WHERE O_ID = %s AND O_D_ID = %s AND O_W_ID = %s;

SELECT SUM(OL_AMOUNT) FROM ORDER_LINE
WHERE OL_O_ID = %s AND OL_D_ID = %s AND OL_W_ID = %s;

DELETE FROM NEW_ORDER WHERE NO_D_ID = %s AND NO_W_ID = %s AND NO_O_ID = %s;
UPDATE ORDERS SET O_CARRIER_ID = %s WHERE O_ID = %s AND O_D_ID = %s AND O_W_ID = %s;
UPDATE ORDER_LINE SET OL_DELIVERY_D = %s WHERE OL_O_ID = %s AND OL_D_ID = %s AND OL_W_ID = %s;
UPDATE CUSTOMER SET C_BALANCE = C_BALANCE + %s WHERE C_ID = %s AND C_D_ID = %s AND C_W_ID = %s;
```

### v1 (1 batch SELECT + N × 5 queries = ~26-51 round-trips)

v1 merged queries 1+2 into a `GROUP BY NO_D_ID` with `MIN(NO_O_ID)` + `JOIN ORDERS` batch (returns at most 10 rows). sumOLAmount stayed as a per-row query. Writes stayed per-row.

```sql
-- Batch SELECT: 1 query (max 10 rows)
SELECT n.NO_D_ID, n.NO_O_ID, o.O_C_ID
FROM (SELECT NO_D_ID, MIN(NO_O_ID) AS NO_O_ID
      FROM NEW_ORDER WHERE NO_W_ID = %s AND NO_O_ID > -1 GROUP BY NO_D_ID) n
JOIN ORDERS o ON o.O_ID = n.NO_O_ID AND o.O_D_ID = n.NO_D_ID AND o.O_W_ID = %s;

-- Per result row: 5 queries (1 SELECT + 4 writes)
SELECT SUM(OL_AMOUNT) FROM ORDER_LINE WHERE OL_O_ID = %s AND OL_D_ID = %s AND OL_W_ID = %s;
DELETE FROM NEW_ORDER WHERE NO_D_ID = %s AND NO_W_ID = %s AND NO_O_ID = %s;
UPDATE ORDERS SET O_CARRIER_ID = %s WHERE O_ID = %s AND O_D_ID = %s AND O_W_ID = %s;
UPDATE ORDER_LINE SET OL_DELIVERY_D = %s WHERE OL_O_ID = %s AND OL_D_ID = %s AND OL_W_ID = %s;
UPDATE CUSTOMER SET C_BALANCE = C_BALANCE + %s WHERE C_ID = %s AND C_D_ID = %s AND C_W_ID = %s;
```

### v2 (1 batch SELECT + 4 batch writes = 5 round-trips)

**KB method: COMBINING_QUERIES** — inlines sumOLAmount as a correlated subquery (safe: GROUP BY MIN limits to ≤10 rows), then replaces all per-row writes with 4 batch statements.

```sql
-- Batch SELECT with inline sumOLAmount: 1 query
SELECT n.NO_D_ID, n.NO_O_ID, o.O_C_ID,
       (SELECT SUM(OL_AMOUNT) FROM ORDER_LINE
        WHERE OL_O_ID = n.NO_O_ID AND OL_D_ID = n.NO_D_ID AND OL_W_ID = o.O_W_ID) AS ol_total
FROM (SELECT NO_D_ID, MIN(NO_O_ID) AS NO_O_ID
      FROM NEW_ORDER WHERE NO_W_ID = %s AND NO_O_ID > -1 GROUP BY NO_D_ID) n
JOIN ORDERS o ON o.O_ID = n.NO_O_ID AND o.O_D_ID = n.NO_D_ID AND o.O_W_ID = %s;

-- Batch DELETE: 1 query (all districts)
DELETE FROM NEW_ORDER WHERE NO_W_ID = %s AND (NO_D_ID, NO_O_ID) IN ((%s,%s), ...);

-- Batch UPDATE ORDERS: 1 query, same carrier for all
UPDATE ORDERS SET O_CARRIER_ID = %s WHERE O_W_ID = %s AND (O_ID, O_D_ID) IN ((%s,%s), ...);

-- Batch UPDATE ORDER_LINE: 1 query, same timestamp for all
UPDATE ORDER_LINE SET OL_DELIVERY_D = %s WHERE OL_W_ID = %s AND (OL_O_ID, OL_D_ID) IN ((%s,%s), ...);

-- Batch UPDATE CUSTOMER: 1 query, CASE per customer for different totals
UPDATE CUSTOMER SET C_BALANCE = C_BALANCE + CASE
    WHEN C_ID = %s AND C_D_ID = %s THEN %s
    ...
    ELSE 0
END WHERE C_W_ID = %s AND ((C_ID = %s AND C_D_ID = %s) OR ...);
```

**Critical detail**: Each batch write includes `NO_W_ID = %s`/`O_W_ID = %s`/`OL_W_ID = %s` as a guard. Without it, `(NO_D_ID, NO_O_ID)` pairs are only unique per-warehouse and would corrupt other warehouses.

| | Baseline | v1 | v2 |
|---|---|---|---|
| Round-trips | 70 | ~26-51 | **5** |
| vs baseline | — | ~1.4-2.7× | **14×** |

## NEW_ORDER

### Application flow

1. For each item (N = 5-15): `SELECT I_PRICE, I_NAME, I_DATA FROM ITEM` — per-item loop
2. `SELECT W_TAX FROM WAREHOUSE`
3. `SELECT D_TAX, D_NEXT_O_ID FROM DISTRICT`
4. `SELECT C_DISCOUNT, C_LAST, C_CREDIT FROM CUSTOMER`
5. `UPDATE DISTRICT SET D_NEXT_O_ID`
6. `INSERT INTO ORDERS` + `INSERT INTO NEW_ORDER`
7. For each item: `SELECT S_QUANTITY, ... FROM STOCK` — per-item loop
8. For each item: `UPDATE STOCK SET ...` — per-item loop
9. For each item: `INSERT INTO ORDER_LINE` — per-item loop

### Baseline (6 + 4N round-trips)

5-15 line items × 4 loops = ~46 round-trips at N=10.

### v1 + v2 (8 round-trips — unchanged from v1, already optimal)

**KB methods applied in both v1 and v2:**
- **COMBINING_QUERIES** — batch item SELECT via `IN` clause, batch stock SELECT via `(I_ID, W_ID) IN`, merge 3 independent SELECTs (warehouse + district + customer) into a 3-way `JOIN`
- **COMBINING_QUERIES** — batch stock UPDATE via `CASE` expression
- **COMBINING_QUERIES** — batch ORDER_LINE INSERT via multi-row `VALUES`

```sql
-- Batch item lookup: 1 query (instead of N)
SELECT I_ID, I_PRICE, I_NAME, I_DATA FROM ITEM WHERE I_ID IN (%s, %s, ..., %s);

-- Merge warehouse + district + customer: 1 query (instead of 3)
SELECT w.W_TAX, d.D_TAX, d.D_NEXT_O_ID, c.C_DISCOUNT, c.C_LAST, c.C_CREDIT
FROM WAREHOUSE w
JOIN DISTRICT d ON d.D_W_ID = w.W_ID AND d.D_ID = %s
JOIN CUSTOMER c ON c.C_W_ID = w.W_ID AND c.C_D_ID = %s AND c.C_ID = %s
WHERE w.W_ID = %s;

-- 3 writes (unchanged)
UPDATE DISTRICT SET D_NEXT_O_ID = %s WHERE D_ID = %s AND D_W_ID = %s;
INSERT INTO ORDERS (...) VALUES (...);
INSERT INTO NEW_ORDER (...) VALUES (...);

-- Batch stock lookup: 1 query (instead of N)
SELECT S_I_ID, S_W_ID, S_QUANTITY, S_DATA, S_YTD, S_ORDER_CNT, S_REMOTE_CNT, S_DIST_%02d
FROM STOCK WHERE (S_I_ID, S_W_ID) IN ((%s,%s), (%s,%s), ...);

-- Batch stock update via CASE: 1 query (instead of N)
UPDATE STOCK SET
  S_QUANTITY = CASE WHEN (S_I_ID, S_W_ID) = (%s,%s) THEN %s ... ELSE S_QUANTITY END,
  S_YTD = CASE ... END,
  S_ORDER_CNT = CASE ... END,
  S_REMOTE_CNT = CASE ... END
WHERE (S_I_ID = %s AND S_W_ID = %s) OR ...;

-- Batch order line insert: 1 query (instead of N)
INSERT INTO ORDER_LINE (...) VALUES (%s,%s,...), (%s,%s,...), ...;
```

| | Baseline (N=10) | v1 / v2 |
|---|---|---|
| Round-trips | 46 | **8** |
| vs baseline | — | **~6×** |

## ORDER_STATUS

### Application flow

Two paths:
- **by c_id**: customer ID known upfront
- **by c_last**: search by last name, pick median customer

Both then find the customer's last order and its line items.

### Baseline (3 round-trips for either path)

```sql
-- Path 1 (by c_id): customer info
SELECT C_ID, C_FIRST, C_MIDDLE, C_LAST, C_BALANCE
FROM CUSTOMER WHERE C_W_ID = %s AND C_D_ID = %s AND C_ID = %s;

-- Path 2 (by c_last): find customers, pick median
SELECT C_ID, C_FIRST, C_MIDDLE, C_LAST, C_BALANCE
FROM CUSTOMER WHERE C_W_ID = %s AND C_D_ID = %s AND C_LAST = %s ORDER BY C_FIRST;

-- Then (both paths):
SELECT O_ID, O_CARRIER_ID, O_ENTRY_D
FROM ORDERS WHERE O_W_ID = %s AND O_D_ID = %s AND O_C_ID = %s ORDER BY O_ID DESC LIMIT 1;

-- If order exists:
SELECT OL_SUPPLY_W_ID, OL_I_ID, OL_QUANTITY, OL_AMOUNT, OL_DELIVERY_D
FROM ORDER_LINE WHERE OL_W_ID = %s AND OL_D_ID = %s AND OL_O_ID = %s;
```

### v1 (by c_id: 2 RTs; by c_last: 3 RTs)

v1 merged customer + last order into one query via `LEFT JOIN` + `ORDER BY ... LIMIT 1`. Order lines stayed as a separate query.

```sql
-- by c_id path: 2 queries
-- Query 1: merged customer + last order
SELECT c.C_ID, c.C_FIRST, c.C_MIDDLE, c.C_LAST, c.C_BALANCE,
       o.O_ID, o.O_CARRIER_ID, o.O_ENTRY_D
FROM CUSTOMER c
LEFT JOIN ORDERS o
  ON o.O_W_ID = c.C_W_ID AND o.O_D_ID = c.C_D_ID AND o.O_C_ID = c.C_ID
WHERE c.C_W_ID = %s AND c.C_D_ID = %s AND c.C_ID = %s
ORDER BY o.O_ID DESC LIMIT 1;

-- Query 2: order lines (if order exists)
SELECT OL_SUPPLY_W_ID, OL_I_ID, OL_QUANTITY, OL_AMOUNT, OL_DELIVERY_D
FROM ORDER_LINE WHERE OL_W_ID = %s AND OL_D_ID = %s AND OL_O_ID = %s;

-- by c_last path: 3 queries
-- Query 1: find customers (same as baseline)
SELECT C_ID, C_FIRST, C_MIDDLE, C_LAST, C_BALANCE
FROM CUSTOMER WHERE C_W_ID = %s AND C_D_ID = %s AND C_LAST = %s ORDER BY C_FIRST;

-- Query 2: last order
SELECT O_ID, O_CARRIER_ID, O_ENTRY_D
FROM ORDERS WHERE O_W_ID = %s AND O_D_ID = %s AND O_C_ID = %s ORDER BY O_ID DESC LIMIT 1;

-- Query 3: order lines (if order exists)
SELECT OL_SUPPLY_W_ID, OL_I_ID, OL_QUANTITY, OL_AMOUNT, OL_DELIVERY_D
FROM ORDER_LINE WHERE OL_W_ID = %s AND OL_D_ID = %s AND OL_O_ID = %s;
```

### v2 (by c_id: 1 RT; by c_last: 2 RTs)

**KB method: COMBINING_QUERIES** — merges order lines into the same query using a derived table (c_id path) or subquery (c_last path). Eliminates the need for a separate `getOrderLines` call.

**by c_id** — single query:
```sql
SELECT c.C_ID, c.C_FIRST, c.C_MIDDLE, c.C_LAST, c.C_BALANCE,
       lo.O_ID, lo.O_CARRIER_ID, lo.O_ENTRY_D,
       ol.OL_SUPPLY_W_ID, ol.OL_I_ID, ol.OL_QUANTITY, ol.OL_AMOUNT, ol.OL_DELIVERY_D
FROM CUSTOMER c
LEFT JOIN (
    SELECT O_ID, O_CARRIER_ID, O_ENTRY_D, O_W_ID, O_D_ID, O_C_ID
    FROM ORDERS
    WHERE O_W_ID = %s AND O_D_ID = %s AND O_C_ID = %s
    ORDER BY O_ID DESC LIMIT 1
) lo ON true
LEFT JOIN ORDER_LINE ol ON ol.OL_W_ID = lo.O_W_ID AND ol.OL_D_ID = lo.O_D_ID AND ol.OL_O_ID = lo.O_ID
WHERE c.C_W_ID = %s AND c.C_D_ID = %s AND c.C_ID = %s
ORDER BY ol.OL_NUMBER;
```

Returns 0-N rows. `rows[0][5]` is `O_ID`: if `None`, no order exists. If not `None`, `rows[0][5:8]` is the order, and all rows with non-null `OL_I_ID` form the order lines.

**by c_last** — same order+lines merge, but customer lookup stays separate (need median):
```sql
-- Query 1: find customers by last name (unchanged)
SELECT C_ID, C_FIRST, C_MIDDLE, C_LAST, C_BALANCE
FROM CUSTOMER WHERE C_W_ID = %s AND C_D_ID = %s AND C_LAST = %s ORDER BY C_FIRST;

-- Query 2: merged order + lines for the picked customer
SELECT O.O_ID, O.O_CARRIER_ID, O.O_ENTRY_D,
       OL.OL_SUPPLY_W_ID, OL.OL_I_ID, OL.OL_QUANTITY, OL.OL_AMOUNT, OL.OL_DELIVERY_D
FROM ORDERS O
LEFT JOIN ORDER_LINE OL
    ON OL.OL_W_ID = O.O_W_ID AND OL.OL_D_ID = O.O_D_ID AND OL.OL_O_ID = O.O_ID
WHERE O.O_W_ID = %s AND O.O_D_ID = %s AND O.O_C_ID = %s
  AND O.O_ID = (SELECT MAX(O2.O_ID) FROM ORDERS O2
                WHERE O2.O_W_ID = O.O_W_ID AND O2.O_D_ID = O.O_D_ID AND O2.O_C_ID = O.O_C_ID)
ORDER BY OL.OL_NUMBER;
```

| | Baseline | v1 | v2 |
|---|---|---|---|
| by c_id | 3 | 2 | **1** |
| by c_last | 3 | 3 | **2** |

## PAYMENT

### Application flow

1. Find customer (by id or last name)
2. `SELECT W_NAME, ... FROM WAREHOUSE`
3. `SELECT D_NAME, ... FROM DISTRICT`
4. `UPDATE WAREHOUSE SET W_YTD`
5. `UPDATE DISTRICT SET D_YTD`
6. `UPDATE CUSTOMER SET C_BALANCE, C_YTD_PAYMENT, C_PAYMENT_CNT`
7. `INSERT INTO HISTORY`

### Baseline (7 round-trips)

```sql
-- Query 1: customer (by id or last name)
SELECT C_ID, C_FIRST, C_MIDDLE, C_LAST, C_STREET_1, C_STREET_2, C_CITY,
       C_STATE, C_ZIP, C_PHONE, C_SINCE, C_CREDIT, C_CREDIT_LIM,
       C_DISCOUNT, C_BALANCE, C_YTD_PAYMENT, C_PAYMENT_CNT, C_DATA
FROM CUSTOMER WHERE C_W_ID = %s AND C_D_ID = %s AND C_ID = %s;

-- Query 2: warehouse (separate)
SELECT W_NAME, W_STREET_1, W_STREET_2, W_CITY, W_STATE, W_ZIP
FROM WAREHOUSE WHERE W_ID = %s;

-- Query 3: district (separate)
SELECT D_NAME, D_STREET_1, D_STREET_2, D_CITY, D_STATE, D_ZIP
FROM DISTRICT WHERE D_W_ID = %s AND D_ID = %s;

-- 4 writes (unchanged across all versions)
UPDATE WAREHOUSE SET W_YTD = W_YTD + %s WHERE W_ID = %s;
UPDATE DISTRICT SET D_YTD = D_YTD + %s WHERE D_W_ID = %s AND D_ID = %s;
UPDATE CUSTOMER SET C_BALANCE = ..., C_YTD_PAYMENT = ..., ... WHERE ...;
INSERT INTO HISTORY VALUES (...);
```

### v1 (by c_id: 6 RTs; by c_last: 7 RTs)

v1 merged warehouse + district into one `JOIN` query, but kept customer separate.

```sql
-- c_id path: 2 SELECTs
-- Query 1: customer (separate, unchanged from baseline)
SELECT C_ID, C_FIRST, C_MIDDLE, C_LAST, C_STREET_1, C_STREET_2, C_CITY, ... FROM CUSTOMER ...;

-- Query 2: merged warehouse + district (1 query instead of 2)
SELECT w.W_NAME, w.W_STREET_1, w.W_STREET_2, w.W_CITY, w.W_STATE, w.W_ZIP,
       d.D_NAME, d.D_STREET_1, d.D_STREET_2, d.D_CITY, d.D_STATE, d.D_ZIP
FROM WAREHOUSE w
JOIN DISTRICT d ON d.D_W_ID = w.W_ID AND d.D_ID = %s
WHERE w.W_ID = %s;

-- c_last path: 3 SELECTs (customer lookup + merged W+D)
SELECT C_ID, C_FIRST, C_MIDDLE, C_LAST, C_STREET_1, C_STREET_2, ... FROM CUSTOMER ... ORDER BY C_FIRST;
-- ... pick median, then same merged warehouse+district query above
```

### v2 (by c_id: 5 RTs; by c_last: 6 RTs)

**KB method: COMBINING_QUERIES** — for the c_id path, merges customer + warehouse + district into a single 3-table comma join:

```sql
-- c_id path: 1 SELECT (instead of 3) — all three tables in one query
SELECT c.C_ID, c.C_FIRST, c.C_MIDDLE, c.C_LAST,
       c.C_STREET_1, c.C_STREET_2, c.C_CITY, c.C_STATE, c.C_ZIP,
       c.C_PHONE, c.C_SINCE, c.C_CREDIT, c.C_CREDIT_LIM,
       c.C_DISCOUNT, c.C_BALANCE, c.C_YTD_PAYMENT, c.C_PAYMENT_CNT, c.C_DATA,
       w.W_NAME, w.W_STREET_1, w.W_STREET_2, w.W_CITY, w.W_STATE, w.W_ZIP,
       d.D_NAME, d.D_STREET_1, d.D_STREET_2, d.D_CITY, d.D_STATE, d.D_ZIP
FROM CUSTOMER c, WAREHOUSE w, DISTRICT d
WHERE c.C_W_ID = %s AND c.C_D_ID = %s AND c.C_ID = %s
  AND w.W_ID = %s
  AND d.D_W_ID = %s AND d.D_ID = %s;

-- c_last path unchanged from v1 (3 SELECTs: customer lookup + merged W+D)
```

The c_last path can't merge customer with warehouse/district — the median customer computation requires a round-trip to fetch all matching last names first.

| | Baseline | v1 | v2 |
|---|---|---|---|
| by c_id | 7 | 6 | **5** |
| by c_last | 7 | 7 | **6** |

## STOCK_LEVEL

### Application flow

1. `SELECT D_NEXT_O_ID FROM DISTRICT` → get current OID
2. `SELECT COUNT(DISTINCT OL_I_ID) FROM ORDER_LINE, STOCK` → count items below threshold in last 20 orders

### Baseline (2 round-trips)

```sql
SELECT D_NEXT_O_ID FROM DISTRICT WHERE D_W_ID = %s AND D_ID = %s;

SELECT COUNT(DISTINCT(OL_I_ID)) FROM ORDER_LINE, STOCK
WHERE OL_W_ID = %s AND OL_D_ID = %s
  AND OL_O_ID < %s AND OL_O_ID >= %s
  AND S_W_ID = %s AND S_I_ID = OL_I_ID AND S_QUANTITY < %s;
```

### v1 (1 round-trip — duplicated subquery + EXISTS)

v1 inlined the `D_NEXT_O_ID` lookup as a subquery but duplicated it for the lower bound (`D_NEXT_O_ID - 20`). Used `EXISTS` instead of comma join.

```sql
SELECT COUNT(DISTINCT(OL_I_ID))
FROM ORDER_LINE
WHERE OL_W_ID = %s
  AND OL_D_ID = %s
  AND OL_O_ID < (SELECT D_NEXT_O_ID FROM DISTRICT WHERE D_W_ID = %s AND D_ID = %s)
  AND OL_O_ID >= (SELECT D_NEXT_O_ID - 20 FROM DISTRICT WHERE D_W_ID = %s AND D_ID = %s)
  AND EXISTS (
    SELECT 1 FROM STOCK
    WHERE S_W_ID = %s AND S_I_ID = OL_I_ID AND S_QUANTITY < %s
  )
```

The `D_NEXT_O_ID` subquery appears twice — once for the upper bound, once (minus 20) for the lower bound.

### v2 (1 round-trip — derived table + JOIN)

**KB methods: COMBINING_QUERIES + PREDICATE_PUSHDOWN** — replaces duplicated subquery with a single-row derived table, and replaces `EXISTS` with a `JOIN` that pushes the threshold filter into the join condition (MySQL 5.7 compatible — no CTE).

```sql
SELECT COUNT(DISTINCT(OL_I_ID))
FROM ORDER_LINE
JOIN STOCK ON S_W_ID = OL_W_ID AND S_I_ID = OL_I_ID AND S_QUANTITY < %s
JOIN (SELECT D_NEXT_O_ID - 20 AS low, D_NEXT_O_ID AS high
      FROM DISTRICT WHERE D_W_ID = %s AND D_ID = %s) b
WHERE OL_W_ID = %s AND OL_D_ID = %s
  AND OL_O_ID < b.high
  AND OL_O_ID >= b.low;
```

The derived table `b` produces exactly 1 row. The `S_QUANTITY < %s` is a `JOIN` condition (predicate pushdown), not a `WHERE` clause — MySQL evaluates it during the join rather than as a post-filter.

| | Baseline | v1 | v2 |
|---|---|---|---|
| Round-trips | 2 | 1 | **1** |
| Subquery duplication | — | 2× D_NEXT_O_ID | **none** (derived table) |
| MySQL version | any | any | any (no CTE) |

# Summary

## Round-trip reduction

| Transaction | Baseline | v1 | v2 | v2 vs baseline |
|---|---|---|---|---|---|
| DELIVERY (10 districts) | 70 | ~26-51 | **5** | **14×** |
| NEW_ORDER (N=10) | 46 | 8 | **8** | ~6× |
| ORDER_STATUS (by c_id) | 3 | 2 | **1** | **3×** |
| ORDER_STATUS (by c_last) | 3 | 3 | **2** | 1.5× |
| PAYMENT (by c_id) | 7 | 6 | **5** | 1.4× |
| PAYMENT (by c_last) | 7 | 7 | **6** | 1.2× |
| STOCK_LEVEL | 2 | 1 | **1** | **2×** |

## KB methods applied in v2

| Method | Application | Transactions |
|---|---|---|
| **COMBINING_QUERIES** | Inline sumOLAmount via correlated subquery | DELIVERY |
| **COMBINING_QUERIES** | Batch 4 per-district writes into 4 batch stmts | DELIVERY |
| **COMBINING_QUERIES** | Merge getLastOrder + getOrderLines via derived table/subquery | ORDER_STATUS |
| **COMBINING_QUERIES** | Merge customer + warehouse + district SELECT via comma join | PAYMENT (c_id) |
| **COMBINING_QUERIES** | Replace EXISTS with JOIN + derived table (eliminate subquery dup) | STOCK_LEVEL |
| **PREDICATE_PUSHDOWN** | S_QUANTITY < threshold as JOIN condition rather than WHERE | STOCK_LEVEL |

## Notes

- **DELIVERY batch writes** must include `NO_W_ID = %s` / `O_W_ID = %s` / `OL_W_ID = %s` — the `(d_id, o_id)` pair is unique only per warehouse.
- **ORDER_STATUS derived table** (c_id path) uses a self-contained subquery for the latest order — no lateral join, works on MySQL 5.7.
- **STOCK_LEVEL** uses a `JOIN` with a single-row derived table instead of a CTE (MySQL 5.7 limitation).
- **NEW_ORDER batch stock update** params must be column-major (all quantities, then all ytds, etc.), not row-major.
